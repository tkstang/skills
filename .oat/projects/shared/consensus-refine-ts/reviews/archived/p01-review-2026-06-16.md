### Important

- **Canonical wrapper source is effectively untyped at the new TypeScript boundary** (`src/consensus/refine/consensus-refine.ts:30`)
  - Issue: Phase p01-t02 and the design call for the wrapper to become canonical TypeScript that type-checks against the loop API with "genuine types where they prevent bugs." The new file defines `AnyRecord = Record<string, any>` and then uses explicit `any` broadly across exported APIs and the wrapper/loop boundary. `pnpm run type-check` passes, but it will not catch option, manifest, resume-state, section-result, or loop-call shape drift in the areas this migration is supposed to make safer.
  - Fix: Add focused local interfaces/types for the wrapper public options, run options, resume state, section records/results, manifest entries, provider inventory entries, and loop invocation payloads. Keep parse/JSON boundaries as `unknown` where appropriate and narrow them before use. This does not need a ground-up retype, but the exported surfaces and `runConsensusLoop` call sites should stop relying on `any`.
  - Requirement: p01-t02

### Minor

- **Import rewrite replaces every matching string literal, not just module specifiers** (`scripts/build-generated.mjs:115`)
  - Issue: The rewrite loop searches the full emitted output and `replaceAll`s quoted occurrences of the source specifier. The current wrapper only contains `../core/consensus-loop.js` in the import, so this works today, but a future diagnostic string or data literal containing the same text would be rewritten silently even though it is not an import specifier.
  - Suggestion: Constrain the rewrite to static module specifiers, such as `from '<specifier>'` / `from "<specifier>"` and bare side-effect import forms, and optionally fail if the number of rewritten import specifiers is not exactly what the mapping expects.

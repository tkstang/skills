### Important

- **No-config convergence defaults can silently replace a built-in peer** (`src/consensus/create/consensus-create.ts:1245`)
  - Issue: `create`, `decide`, `plan`, and `evaluate` load provider inventory when explicit `--peers` is absent, then map the resolver's built-in convergence agents directly into `peers`. The resolver's built-in branch uses `builtInAgents(input.inventory, 2)`, and `builtInAgents` filters to ready providers before slicing. With no config and default inventory `claude,codex,cursor`, if `codex` is `auth_required` while `cursor` is ready, the wrappers choose `claude,cursor` and continue instead of preserving the old `claude,codex` built-in peer order and failing preflight on `codex`. I reproduced this against the generated p02 wrapper outputs; `create`, `decide`, `plan`, and `evaluate` returned `["claude","cursor"]` under that no-config/auth-required setup. `refine` avoids this because it converts built-in resolver results back to its existing host-aware default path.
  - Fix: Keep convergence built-ins independent from inventory readiness. For example, make `resolveConvergenceComposition` return `claude,codex` for built-in convergence regardless of inventory, or have these wrappers treat `composition.source === 'built-in'` the way `refine` does and use their previous explicit built-in defaults before preflight. Add regression coverage where no config exists, a built-in peer is unavailable/auth-required, and another provider is ready; the expected behavior should preserve the built-in pair and fail preflight rather than substituting the other provider.
  - Requirement: p02 no-config behavior preserves built-in peer order.

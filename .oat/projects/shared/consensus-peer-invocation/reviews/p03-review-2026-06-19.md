## Summary

p03 adds the provider CLI invocation seam, provider-neutral audit fields, retry-boundary changes, Refine/Evaluate dogfood switches, integration tests, generated runtime updates, and smoke coverage. The focused p03 tests, type-check, build drift check, and default smoke pass, but the provider CLI backend is not actually usable through its default generated CLI path, and Refine prepared-parallel runs silently fall back to the old backend even when the provider CLI backend is selected.

## Findings

### Critical

- **Default generated provider CLI path is spawned as a non-executable file** (`src/consensus/core/consensus-loop.ts:1414`)
  - Issue: `resolveConsensusCliPath()` resolves the default backend command to `plugins/consensus/scripts/consensus.mjs` from generated runtime, and `runProviderCliCommand()` spawns that path directly. The generated CLI is committed as mode `100644`, so selecting `CONSENSUS_PROVIDER_BACKEND=provider-cli` without a custom executable `CONSENSUS_CLI_PATH` fails with `EACCES` before provider inventory or preflight can run. This breaks the p03 dogfood switch's generated CLI resolution path; the tests only cover executable fixture overrides.
  - Evidence: `node --input-type=module` calling generated `preflightConsensusProviderCli({ env: { ...process.env, CONSENSUS_PROVIDER_BACKEND: 'provider-cli' } })` failed with `spawn .../plugins/consensus/scripts/consensus.mjs EACCES`.
  - Fix: Either make generated `plugins/consensus/scripts/consensus.mjs` executable through the build pipeline and git mode, or have the invoker execute generated `.mjs` files via `process.execPath <script> ...` while preserving direct execution for explicit executable overrides. Add a test that selects `provider-cli` without `CONSENSUS_CLI_PATH` against the generated runtime.
  - Requirement: FR4, FR8

- **Prepared parallel Refine packets ignore the provider CLI backend switch** (`src/consensus/refine/consensus-refine.ts:3293`)
  - Issue: `prepareParallelRun()` preflights the selected backend, but the emitted section packet only carries `loop_argv`. When those section runners execute `consensus-loop.mjs`, `runConsensusLoop()` still defaults to `invokeValidatedPeer()` / `invokePaseoWithRetry()`, so a run prepared with `CONSENSUS_PROVIDER_BACKEND=provider-cli` executes section work through the old backend and writes `raw_paseo_response` instead of the p03 provider-neutral audit fields.
  - Evidence: A live prepare-parallel probe with `makeProviderCliEnv()` produced a section record containing `raw_paseo_response` and no `raw_provider_response` after executing the emitted `loop_argv`.
  - Fix: Propagate backend selection into section runner execution. A robust fix is to make the core loop default invoker honor the same provider backend env/options, or include explicit provider-backend metadata in packets and section-runner dispatch so `loop_argv` execution uses `invokeProviderCliWithRetry()` / `invokeConsensusProviderCli()`. Add a prepared-parallel provider CLI integration test that asserts section records contain provider-neutral audit fields.
  - Requirement: FR4, FR7

### Important

- **Refine provider CLI preflight does not check an explicit synthesizer outside the peer pair** (`src/consensus/refine/consensus-refine.ts:3764`)
  - Issue: `preflightConsensusProviderCli()` resolves and preflights only `resolved.peers`. In `parallel_synthesized`, an explicit synthesizer such as `cursor` can be present in provider inventory but never receive the selected-provider `consensus preflight --json --provider cursor` check before the run. Evaluate correctly preflights the union of peers and synthesizer; Refine should match that behavior.
  - Fix: Include the resolved synthesizer in provider CLI selected-provider preflight when `parallel_synthesized` is active and the synthesizer is outside the peer set. Add a Refine test where inventory reports the synthesizer but its per-provider preflight is unusable, and assert the wrapper fails before running the loop.
  - Requirement: FR5

### Important

- **Output cap reports an error but still returns oversized captured output** (`src/consensus/provider-cli/subprocess.ts:95`)
  - Issue: `runProviderSubprocess()` appends each stdout/stderr chunk before checking `stdoutBytes + stderrBytes > maxOutputBytes`, and the failure result returns the full accumulated `stdout`/`stderr`. A targeted probe against the p02 fixture returned `PROVIDER_OUTPUT_CAP_EXCEEDED` for a 16-byte cap while retaining 1024 bytes of stdout and diagnostics `stdoutBytes: 1024`. This violates the p02 requirement for bounded buffers/runaway-output protection and means a provider can still force the parent to retain output beyond the configured cap.
  - Fix: Bound captured buffers before appending beyond the remaining budget, stop reading or destroy stdout/stderr once the cap is reached, and make the output-cap termination path avoid retaining more provider output while waiting for process shutdown. Add a regression test asserting returned stdout/stderr length and diagnostics never exceed the configured cap, including a child that keeps writing after SIGTERM.

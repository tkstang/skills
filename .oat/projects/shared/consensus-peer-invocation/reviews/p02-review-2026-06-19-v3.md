## Findings

### Critical

- **Generated inventory and preflight never run the implemented provider probes** (`src/consensus/provider-cli/commands.ts:244`)
  - Issue: `resolveRegistry()` only calls `probeProviderRegistry()` when a test or caller injects `options.probeRunner`; otherwise it falls back to `defaultProviderRegistry()`, which marks every first-scope provider as `missing`. The generated process entrypoint calls `runConsensusCli(process.argv.slice(2), nodeIo())` without supplying `nodeProbeCommandRunner(io.env)`, so the real CLI does not use the p02 readiness probes. On this machine, `which codex` returns `/opt/homebrew/bin/codex` and `which claude` returns `/Users/tstang/.local/bin/claude`, but `node plugins/consensus/scripts/consensus.mjs provider ls --json` reports both as `status:"missing"` and `preflight --provider codex` reports `usable:false`.
  - Fix: Wire the default command path to `nodeProbeCommandRunner(io.env)` for `provider ls` and `preflight` when no explicit registry/probe runner is supplied. Add a generated CLI process test with a temp PATH fixture proving an executable provider is reported `ready`, while absent providers remain `missing`.

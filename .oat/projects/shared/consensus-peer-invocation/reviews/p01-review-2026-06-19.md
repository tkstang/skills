  - Issue: The design specifies `--permission-mode`, `--sandbox`, `--approval-policy`, `--env-allow`, and `--max-depth` as optional `consensus run` flags, and also lists the runtime-policy flags in the `--request-json` conflict contract (`design.md:485`, `design.md:487`). The current `parseRunCommand` allowlist only accepts `--json`, provider/schema/prompt/request-json, max attempts/runtime/output bytes, model, effort, and cwd (`src/consensus/provider-cli/args.ts:186`). The shipped generated CLI therefore returns `CONSENSUS_CLI_USAGE` for design-valid invocations such as `consensus run ... --permission-mode read-only` and `consensus run ... --max-depth 2`.
  - Impact: This leaves the public p01 CLI contract incomplete and blocks later host-guard/runtime-policy work from being driven through the CLI without revisiting the parser and generated entrypoint. Because p01 is the declared CLI argument-surface phase, this should be fixed before accepting the phase.
  - Fix: Add the missing run flags to `ParsedRunCommand`, `parseRunCommand`, conflict detection, `normalizeRunRequest`, and `helpText`; map runtime-policy flags into `ConsensusCliRunRequest.runtime_policy`; carry depth according to the host-context design or explicitly revise `design.md`/`plan.md` if `--max-depth` is intentionally preflight-only. Add unit and process tests, then regenerate `plugins/consensus/scripts/consensus.mjs`.

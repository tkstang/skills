## Summary

Phase 2 adds the expected provider adapter registry, readiness probes, host guard helpers, runtime-policy validation, invocation builders, bounded subprocess runner, generated CLI output, and focused tests. The existing p02 tests, type-check, and generated-output check pass when run directly against the provider-cli subset, and the p02-t07 update to `tests/consensus/provider-cli/cli-process.test.ts` is acceptable because that task's verification explicitly includes the process contract once `run` is implemented.

The implementation is not ready to pass: the `run` path can miss native host runtime markers and fail to propagate recursion-depth metadata, and several provider execution safety requirements are only partially enforced.

## Findings

### Critical

- **Run command can bypass same-host recursion tracking** (`src/consensus/provider-cli/args.ts:380`)
  - Issue: `normalizeHostContext()` derives `runtime` only from `CONSENSUS_PARENT_HOST`; it ignores the Claude/Codex/Cursor environment markers that `detectHostRuntime()` handles. A first-level `consensus run --provider codex --max-depth 1` launched from Codex without `CONSENSUS_PARENT_HOST` is normalized as `unknown`, so `evaluateHostGuard()` allows it with `guard: none` and no `CONSENSUS_DEPTH` child env. That defeats the p02 host guard on the actual `run` path and can leave recursive same-host subprocesses untracked.
  - Fix: Reuse `hostContextFromEnv()` or `detectHostRuntime()` when normalizing run host context, and add a CLI/process test showing native host markers produce `same_host` / `subprocess_isolated` at depth 0 and `HOST_RECURSION_BLOCKED` at depth 1.
  - Requirement: FR9, NFR2

### Important

- **Default non-interactive runtime policy is validated but not enforced on provider invocations** (`src/consensus/provider-cli/structured-output.ts:65`)
  - Issue: `validateProviderOptions()` applies `defaultRuntimePolicy()`, but the effective policy is not written back to the request passed into `buildProviderInvocation()`. Invocation builders only emit explicit request policy fields, and `permission_mode` is not mapped at all. As a result, a default `consensus run` can still launch a provider with that provider's interactive/default posture, contrary to the p02 requirement that the default policy be non-interactive and not wait for approval prompts.
  - Fix: Compute an effective runtime policy once, pass it into invocation construction, and have adapters map supported non-interactive policy to concrete argv/env behavior. Add tests that default runs include the non-interactive provider controls or fail if the adapter cannot enforce them.
  - Requirement: NFR5, NFR2

- **Child environment passes unrelated provider credentials to every provider** (`src/consensus/provider-cli/runtime-policy.ts:54`)
  - Issue: `PROVIDER_ENV_ALLOWLIST` contains Anthropic, Claude, OpenAI, and Cursor credentials, and `buildChildEnvironment()` includes the whole list for every provider. A Cursor subprocess can receive OpenAI/Anthropic secrets and a Claude subprocess can receive Cursor/OpenAI secrets. This violates the design's provider-required-only environment boundary and the p02 test intent to exclude unrelated secrets.
  - Fix: Move provider environment requirements into adapter capabilities or a provider-keyed allowlist, and build the child environment from the selected provider plus explicit caller `env_allowlist`. Add tests for each first-scope provider proving other providers' credentials are omitted.
  - Requirement: NFR5

- **Nonzero provider exits are retried without adapter terminal classification** (`src/consensus/provider-cli/subprocess.ts:163`)
  - Issue: Every nonzero subprocess exit becomes retryable `PROVIDER_EXIT`. The p02 plan and design require timeout/nonzero exit retry classification to follow adapter decisions and explicitly exclude terminal cases such as auth-required or unsupported options. Current behavior can spend the full CLI attempt budget retrying known-terminal provider failures and delays actionable remediation.
  - Fix: Add an adapter-owned run-failure classifier, reuse safe auth/unavailable pattern matching where appropriate, and return terminal `PROVIDER_AUTH_REQUIRED`, `PROVIDER_UNSUPPORTED_OPTION`, or non-retryable `PROVIDER_EXIT` before retrying. Cover terminal and transient nonzero exits in structured-output tests.
  - Requirement: FR6, NFR2

- **Timeout and output-cap handling can hang if a child ignores SIGTERM** (`src/consensus/provider-cli/subprocess.ts:75`)
  - Issue: The runner sets a terminal reason and sends `SIGTERM`, but it only resolves on `close`. A provider process that ignores SIGTERM after a timeout or output-cap breach can keep the promise open indefinitely, so `max_runtime_sec` is not a hard bound for hung provider subprocesses.
  - Fix: Add a bounded escalation path after SIGTERM, such as a short grace timer followed by SIGKILL and a final timeout resolution guard. Add a stub test that ignores SIGTERM and still returns `PROVIDER_TIMEOUT` within the expected bound.
  - Requirement: NFR2, NFR5

# Design: docs-ia

> Lightweight design (quick mode). Scope is the **target information
> architecture** and the execution shape for standing up the docs site and
> slimming the README. The exhaustive page-by-page content mapping is delegated
> to `oat docs analyze` / `oat docs apply`, curated against the IA defined here.

## Overview

Stand up a **Fumadocs** docs app for this repo and migrate the dense reference
content out of `README.md` into a navigable site, leaving the README as a lean
entry point (project description + install matrix + links into the site).

The work is **two-phase**: (A) the operator scaffolds the app via the
user-invocable `/oat-docs-bootstrap` skill (which the agent cannot auto-invoke —
it is `disable-model-invocation`); the agent gathers inputs and verifies the
build is green afterward. (B) the agent runs `oat docs analyze` / `oat docs
apply` to migrate and curate content against the agreed IA, slims the README,
and re-verifies the install matrix.

The **organizing principle**, decided collaboratively, is an **audience split at
the top level**: a **User Guide** trunk (how to install / use / configure) and an
**Engineering** trunk (how it works / how to contribute / why). Subjects that
have both consumer and contributor audiences — Consensus and the standalone
Skills — appear in **both** trunks, with content written for that trunk's
audience. This mirrors the proven structure of the sibling repo
`/Users/tstang/Code/vox/gizmo-slack-app` (also Fumadocs, also `user-guide/` +
`engineering/`), adapted down to this repo's smaller surface.

## Architecture

### Information Architecture (the deliverable)

Two top-level trunks under `docs/`, each with its own `index.md` landing. The
User Guide is flat (task/subject pages); Engineering is subdivided into
sub-section folders, each with its own `index.md` — the same shape as the gizmo
reference.

```
docs/
  index.md                              — Overview: what the repo ships + how to navigate
                                          (AUTOGENERATED by `oat docs generate-index`)

  user-guide/                           ← USER TRUNK (install / use / configure)
    index.md                            — consumer at-a-glance
    installation.md                     — per-provider install matrix (+ prerequisites)  [tag-time gate]
    consensus.md                        — overview & capabilities → links to the (untouched) plugin README
    session-observer.md                 — usage, watch mode, permissions, limitations
    export-session-transcript.md        — modes, sanitization, usage

  engineering/                          ← ENGINEERING TRUNK (how it works / contribute / why)
    index.md
    architecture/
      index.md
      transcript-core.md                — single source of truth + generated copies
      generated-runtime.md              — canonical TS → committed .mjs build contract
    repository-layout.md                — canonical structural reference (repo + package layout)
    contributing/
      index.md
      development.md                    — verification commands (type-check/test/build:check/validate/smoke)
      conventions.md                    — commits, versioning, lint/format, hooks (seed; deeper = link)
    decisions.md                        — "why" / DR rationale — FUTURE SLOT (defined, not fabricated)
```

### What we borrow from the gizmo reference

- **Two-trunk audience split** (`user-guide/` + `engineering/`), each with an
  `index.md` landing.
- **Engineering subdivided** into sub-section folders (architecture,
  contributing), each with its own `index.md`.
- The **generated `index.md` + nested `## Contents` nav contract**
  (`oat docs generate-index`), regenerated on `predev`/`prebuild`; `meta.json`
  per folder controls nav order.

### Where we deliberately diverge from gizmo

- **Installation lives in the User trunk.** Gizmo puts "Getting Started" under
  Engineering because its users live in Slack and never install anything. Our
  users install a CLI/plugin, and the install matrix is the tag-time gate — so
  it belongs in the User trunk. Engineering's "getting started" is
  *contributing/development*.
- **No `operations/` / `integrations/` / `agent-internals/`** — this repo has no
  deployment or runtime surface; those gizmo sections have no analog.
- **Consensus is one user-trunk overview page** that links into the plugin's own
  `plugins/consensus/README.md` (decision B below), not a duplicated usage tree.
- **Engineering → Consensus internals is future fill** — that content does not
  exist in any README today; the consensus-family project documents itself in
  later. We do not pre-create an empty page.

## Component Design

The execution decomposes into four components (phases in the plan).

### A. Scaffold (operator-run + agent verify)

**Purpose:** Stand up the Fumadocs app.
**Responsibilities:**

- Agent **gathers + pre-stages inputs** for the operator's bootstrap run:
  framework `fumadocs`; a real **site display title** (the worktree dir is
  `docs-ia` but the project is `skills` — the humanized-basename default is
  wrong); target dir (single-package → `nested-standalone`; gizmo uses
  `documentation/`, default would be `docs/`); description; `lint`/`format`.
- **Operator runs `/oat-docs-bootstrap`** (agent cannot auto-invoke it).
- Agent **verifies the build is green** afterward and records the **framework
  decision** (Fumadocs) as a DR candidate.

**Design decisions:**

- **B — leave `plugins/consensus/README.md` untouched.** It is already a ~95%
  consumer quick-start; the only "technical detail" is its short "Package
  Layout" section. Deep consensus docs are better landed by the consensus-family
  project documenting itself into the site. This project's blast radius is the
  *root* README + the site scaffold.

### B. Migrate + curate IA

**Purpose:** Move dense reference content into the site under the two-trunk IA.
**Responsibilities:**

- Run `oat docs analyze` to propose structure; **curate the proposal against the
  IA above** (it is tool-assisted curation, not from-scratch design).
- Run `oat docs apply` to write pages.
- Maintain a **no-info-loss mapping**: every block removed from `README.md` has a
  destination page before the README is slimmed.

### C. Slim README + preserve the install matrix

**Purpose:** Reduce `README.md` to an entry point.
**Responsibilities:**

- Final README = project description + **install matrix** + links into the site.
- **Do not rewrite install commands** during the move — relocate/link only.
- **Re-verify the install matrix against the live provider CLIs** (`claude`,
  `codex`, `cursor`) so the `bl-d85f` tag-time gate is not regressed.

### D. Repository-layout page + closeout

**Purpose:** Author the canonical structural reference and update bookkeeping.
**Responsibilities:**

- Author `engineering/repository-layout.md` as the **canonical structural
  reference** (repo layout from the root README, plus a description of the
  consensus package structure) — **without editing the plugin README**.
- Closeout: bl-ecaa status; `completed.md`; `current-state.md` (note the docs
  site + its location); `roadmap.md`; record the **Fumadocs** decision in
  `decision-record.md`.

## Testing Strategy

Verification is build/contract/accuracy oriented (no application unit tests —
this is a docs + content migration project).

| Check | How | Gate |
| --- | --- | --- |
| Docs build green | Operator/agent runs the scaffolded install + build (e.g. `pnpm --filter <app> build` or `cd <app> && pnpm build`) | Phase A acceptance |
| Generated nav/index contract | `oat docs generate-index` produces `index.md`; nested `## Contents` matches the page tree; no drift on `predev`/`prebuild` | Phase B + closeout |
| Repo validate/build | `pnpm run validate` (and `pnpm run build:check` if generated outputs touched) pass | Closeout |
| Install-matrix accuracy | Re-verify README matrix against live `claude` / `codex` / `cursor` CLIs per `RELEASING.md` | Phase C — **must not regress bl-d85f** |
| No information loss | Section-by-section mapping: every removed README block resolves to a live site page | Phase C before slimming |
| Link integrity | Internal links/anchors resolve; README links into the site resolve | Closeout |

## Open Questions

Resolved during the operator's `/oat-docs-bootstrap` run (inputs, not blockers):

- **Site display title** — operator sets a real title (not the `docs-ia`
  worktree basename).
- **Target directory** — `docs/` (single-package default) vs a `documentation/`
  convention like the gizmo reference; operator confirms at scaffold.
- **`decisions.md` slot** — defined now, filled later (lean: do not fabricate a
  stub; the trunk slot exists for future `oat-project-document` runs).

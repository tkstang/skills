# Design: docs-ia

> Lightweight design (quick mode). Scope is the **target information
> architecture** and the execution shape for standing up the docs site and
> slimming the README. The exhaustive page-by-page content mapping is delegated
> to `oat docs analyze` / `oat docs apply`, curated against the IA defined here.

## Overview

Stand up a **Fumadocs** docs app for this repo and migrate the dense reference
content out of `README.md` into a navigable site, leaving the README a lean
entry point (project description + install matrix + links into the site).

The work is **two-phase**: (A) the operator scaffolds the app via the
user-invocable `/oat-docs-bootstrap` skill (which the agent cannot auto-invoke —
it is `disable-model-invocation`); the agent gathers inputs and verifies the
build is green afterward. (B) the agent runs `oat docs analyze` / `oat docs
apply` to migrate and curate content against the agreed IA, slims the README,
and re-verifies the install matrix.

The **organizing principle**, decided collaboratively, is an **audience split at
the top level**: a **User Guide** trunk (how to install / use / configure) and an
**Engineering** trunk (how it works / how to contribute / why). Subjects that
have both audiences — Consensus and the standalone Skills — appear in **both**
trunks, written for that trunk's audience. The structure follows the proven
patterns of three sibling Fumadocs sites — `gizmo-slack-app`, `stoa`, and
`open-agent-toolkit` — adapted to this repo's surface.

## Architecture

### Information Architecture (the deliverable)

Two top-level trunks under `docs/`, each with its own `index.md`. The User Guide
groups each subject in a folder (overview `index.md` + deeper per-item pages);
Engineering is subdivided into `architecture/` and a two-part `contributing/`
(`development/` + `documentation/`).

```
docs/
  index.md                                  — Overview: what the repo ships + how to navigate
                                              (AUTOGENERATED by `oat docs generate-index`)

  user-guide/                               ← USER TRUNK (install / use / configure)
    index.md                                — consumer at-a-glance: what's available
    installation.md                         — per-provider install matrix (+ prerequisites)  [tag-time gate]
    consensus/
      index.md                              — what consensus is; refine vs evaluate; modes/scope; limitations
      refine.md                             — refine: usage, examples, arguments, iteration modes, resume, escalation
      evaluate.md                           — evaluate: usage, examples, arguments, rubric flow, dissent
      configuration.md                      — peers, provider floor, synthesizer, agency, Cursor auth (shared)
    skills/
      index.md                              — what standalone skills are available
      session-observer.md                   — usage, examples, watch mode, arguments, permissions, limitations
      export-session-transcript.md          — usage, examples, modes/flags, sanitization

  engineering/                              ← ENGINEERING TRUNK (how it works / contribute / why)
    index.md
    architecture/
      index.md
      transcript-core.md                    — single source of truth + generated copies
      generated-runtime.md                  — canonical TS → committed .mjs build contract
    repository-layout.md                    — canonical structural reference (repo + package layout)
    contributing/
      index.md                              — routes to development + documentation
      development/
        index.md                            — dev workflow + verification commands (type-check/test/build:check/validate/smoke)
        conventions.md                      — Node-stdlib / dependency-free shipped code, generated-output discipline, pnpm dev deps
        commit-conventions.md               — Conventional Commits + commit-msg hook
        hooks-and-safety.md                 — git hooks, lint-staged, skill version-bump enforcement
      documentation/
        index.md                            — docs authoring contract overview
        authoring.md                        — navigation/index contract, local docs workflow, generated-index discipline
        markdown-features.md                — supported markdown/MDX: frontmatter, callouts, tabs, Mermaid, code blocks
        review-checklist.md                 — docs-change review checklist
    decisions.md                            — "why" / DR rationale — FUTURE SLOT (defined, not fabricated)
```

### What we borrow from the reference sites

- **Two-trunk audience split** (`user-guide/` + `engineering/`), each with an
  `index.md` landing — gizmo / stoa.
- **Subject folders with `index.md` + deeper pages** (consensus/, skills/) — stoa
  `using-stoa/`, `getting-started/`.
- **Agent-grade `contributing/`** split into `development/` + `documentation/`,
  with a **`markdown-features.md`** breakdown and a docs authoring contract —
  oat-docs `contributing/` + stoa `engineering/contributing/`.
- The **generated `index.md` + nested `## Contents` nav contract**
  (`oat docs generate-index`), regenerated on `predev`/`prebuild`; `meta.json`
  per folder controls nav order.

### Where we deliberately diverge / scope down

- **Installation lives in the User trunk** (our users install a CLI/plugin and
  the install matrix is the tag-time gate); engineering's "getting started" is
  *contributing/development*.
- **No `operations/` / `integrations/` / `reference/` trunks** — this repo has
  no deployment/runtime surface or large CLI-reference surface yet; those
  stoa/gizmo sections have no analog. Reference-style material (flags, provider
  CLI commands) lives inside the relevant skill pages, not a separate trunk.
- **Engineering → Consensus internals is future fill** — does not exist in any
  README today; the consensus-family project documents itself in later. Not
  pre-created.

### Scope notes (consequences of the deeper IA)

- **Consensus depth now overlaps `plugins/consensus/README.md`.** The site's
  `user-guide/consensus/` becomes the canonical *rich* user home (full refine /
  evaluate pages). **Decision B still holds** — we do **not** edit the plugin's
  packaged README in this project; it remains a lightweight packaged quick-start.
  A future cleanup (the consensus-family doc step) can trim it to point at the
  site. The overlap is intentional, not accidental.
- **`contributing/documentation/` is partly authored fresh** (markdown-features,
  authoring contract, review-checklist) rather than migrated from the README —
  adapted from the reference sites + the scaffolded `contributing.md` (FP-17) +
  the `oat-docs-authoring` skill. This is a deliberate expansion beyond pure
  README migration to make the site agent-grade from day one.

## Component Design

Four execution components (phases in the plan).

### A. Scaffold (operator-run + agent verify)

**Purpose:** Stand up the Fumadocs app.
**Responsibilities:** agent pre-stages bootstrap inputs (framework `fumadocs`;
real site title — not the `docs-ia` worktree basename; target dir; description;
lint/format); **operator runs `/oat-docs-bootstrap`** (agent cannot auto-invoke
it); agent **verifies the build is green** and records **Fumadocs** as a DR
candidate.
**Design decision B:** leave `plugins/consensus/README.md` untouched — this
project's blast radius is the *root* README + the site.

### B. Migrate + curate IA

**Purpose:** Move dense reference content into the site under the two-trunk IA.
**Responsibilities:** run `oat docs analyze`; **curate against the IA above**;
run `oat docs apply` to write pages; author the deeper subject folders
(`consensus/{index,refine,evaluate,configuration}`,
`skills/{index,session-observer,export-session-transcript}`) and the agent-grade
`contributing/{development,documentation}` subsections; maintain a
**no-info-loss mapping** (every removed README block has a destination before the
README is slimmed).

### C. Slim README + preserve the install matrix

**Purpose:** Reduce `README.md` to an entry point.
**Responsibilities:** final README = description + **install matrix** + links
into the site; **do not rewrite install commands** (relocate/link only);
**re-verify the install matrix against the live provider CLIs** so the bl-d85f
tag-time gate is not regressed.

### D. Repository-layout page + closeout

**Purpose:** Author the canonical structural reference + update bookkeeping.
**Responsibilities:** author `engineering/repository-layout.md` (repo + consensus
package structure) **without editing the plugin README**; closeout (bl-ecaa
status, `completed.md`, `current-state.md` with site location, `roadmap.md`,
Fumadocs DR in `decision-record.md`).

## Testing Strategy

Verification is build/contract/accuracy oriented (no application unit tests —
this is a docs + content migration project).

| Check | How | Gate |
| --- | --- | --- |
| Docs build green | resolve `documentation.root` from `.oat/config.json`; `(cd <app> && pnpm install && pnpm run build)` | Phase A acceptance |
| Generated nav/index contract | `oat docs generate-index` produces `index.md`; nested `## Contents` matches the page tree; no drift on `predev`/`prebuild`; links `.md`-suffixed (FP-16) | Phase B + closeout |
| Repo validate/build | `pnpm run validate` (and `pnpm run build:check` if generated outputs touched) pass | Closeout |
| Install-matrix accuracy | re-verify README matrix against live `claude` / `codex` / `cursor` CLIs per `RELEASING.md` | Phase C — **must not regress bl-d85f** |
| No information loss | section-by-section mapping: every removed README block resolves to a live site page | Phase C before slimming |
| Link integrity | internal links/anchors resolve; README links into the site resolve | Closeout |

# Design: public-discovery

## Overview

This project controls which `SKILL.md` files in the repo are presented as
individually installable skills, across three skill categories that each need a
different treatment. The unifying constraint: we control the `npx skills` CLI
discovery path (it reads our frontmatter and honors `metadata.internal: true`)
but not the skills.sh hosted platform, so the design applies the lever we control
per category and treats skills.sh behavior as something to *verify*, not assume.

The three categories resolve to three independent workstreams. **Standalone
skills** (`session-observer`, `export-session-transcript`) already are the
intended public entries — this category is verification-only. **OAT tooling
skills** (`.agents/skills/**`) are regenerated by the external `oat sync` from
the `open-agent-toolkit` source, so the `internal` flag cannot live in this repo.
This project's concrete cat-3 deliverable is an upstream handoff prompt; the
actual hiding *outcome* (the flag surviving `oat sync` and the skills dropping
from `npx skills` discovery) is **explicitly deferred** to a post-upstream
verification, because it depends on the upstream change landing and being synced
back — outside this project's commit boundary (see Component Design and the
updated discovery success criteria). **Consensus plugin skills** (`refine`,
`evaluate`, `decide`, `plan`, `create`) stay discoverable but are made
*recoverable* when installed standalone: a centralized resolver gains a shared
fallback location, the missing-CLI error becomes actionable, and a repo
`install.sh` (obtained via a pinned fetch from the repo) provisions the shared
script so a standalone-installed skill can be made to work without the full
plugin.

The consensus work is the only code change in this repo, and its blast radius is
small: all five skills resolve the shared CLI through a single function
(`resolveConsensusCliPath`), so the fallback and the actionable error are added
once in canonical TypeScript and inherited by the whole family.

## Architecture

### System Context

Two discovery surfaces and one runtime resolution path interact:

- **`npx skills` CLI** crawls a fixed list of container dirs in the repo
  (`skills/`, `.agents/skills/`, plus plugin-manifest discovery for
  `plugins/consensus`). It honors `metadata.internal: true` (hidden unless
  `INSTALL_INTERNAL_SKILLS=1`). This is the surface we control.
- **skills.sh hosted index** — Vercel-controlled; crawl-vs-submission and whether
  it honors `internal` are unverified. The repo is not yet indexed.
- **Consensus runtime resolution** — at run time each consensus skill locates the
  shared `consensus.mjs` CLI via `resolveConsensusCliPath`. A plugin install
  preserves the whole plugin tree so the existing plugin-relative path resolves;
  a standalone install has only the skill dir, so resolution fails today.

**Key Components:**

- **Category 1 — Standalone skills:** `skills/session-observer`,
  `skills/export-session-transcript`. Self-contained; intended public entries.
  Treatment: **verify only.**
- **Category 2 — Consensus skills:** `plugins/consensus/skills/{refine,evaluate,decide,plan,create}`,
  canonical runtime in `src/consensus/**`, shared CLI at
  `plugins/consensus/scripts/consensus.mjs`. Treatment: **stay discoverable +
  recoverable standalone** (resolver fallback, actionable error, `install.sh`).
- **Category 3 — OAT tooling skills:** `.agents/skills/**` (~56), regenerated by
  external `oat sync`/`oat tools install` from `open-agent-toolkit`. Treatment:
  **upstream handoff prompt** to add `metadata.internal: true` at the pack source.

### Data Flow

Standalone consensus skill run, post-change:

1. User installs one skill via `npx skills add tkstang/skills@refine` (skill dir only).
2. Skill invokes `node ./scripts/consensus-refine.mjs …`, which calls
   `resolveConsensusCliPath`.
3. Resolver checks: explicit arg → `CONSENSUS_CLI_PATH` → plugin-relative path
   (absent) → `~/.consensus/consensus.mjs` (absent on first run) → none found.
4. Runtime emits the actionable missing-CLI message **from the shared core**
   (not a refine-only path): install the consensus plugin, **or** run the pinned
   `install.sh` one-liner (see README).
5. User runs the one-liner; `install.sh` fetches the pinned `consensus.mjs` from
   the repo ref and writes it to `~/.consensus/`.
6. Next run: resolver finds `~/.consensus/consensus.mjs` and proceeds normally.

## Component Design

### resolveConsensusCliPath (shared-space fallback)

**Location:** `src/consensus/core/consensus-loop.ts` (canonical TS) → built to
the generated `.mjs` runtimes via `pnpm run build`.

**Purpose:** Resolve the shared `consensus.mjs` CLI for all five skills.

**Change:** Insert a shared-space lookup into the existing resolution order, after
the plugin-relative default and before declaring the CLI missing:

```
explicit consensusCliPath
  → env.CONSENSUS_CLI_PATH
  → plugin-relative ../../../scripts/consensus.mjs   (if it exists)
  → ~/.consensus/consensus.mjs                        (NEW; if it exists)
  → (none) → caller surfaces actionable missing-CLI error
```

**Design decisions:**

- **Existence-aware resolution.** Today the function returns the plugin-relative
  path unconditionally (string only). To add a fallback that's only used when the
  plugin path is absent, resolution must check existence and fall through. Keep
  `CONSENSUS_CLI_PATH` and explicit arg as highest-priority overrides (unchanged).
- **`~/.consensus/` shared space** — fixed literal `path.join(os.homedir(), '.consensus', 'consensus.mjs')`.
  Chosen over XDG (`~/.local/share/consensus/`) because (a) the toolchain already
  uses home-root dotdirs (`~/.agents`, `~/.claude`, `~/.cursor`, `~/.codex`) and
  (b) a single literal path can't drift between the resolver and `install.sh` (no
  `$XDG_DATA_HOME` branching to keep in sync) — drift is the silent-breakage risk.
- **Dependency-free / Node stdlib only** (`os`, `path`, `fs`) per repo runtime
  contract.

### Actionable missing-CLI error (shared core)

**Location:** the **shared core** provider resolution/invocation surface in
`src/consensus/core/consensus-loop.ts` — the path all five wrappers already go
through (`resolveConsensusCliPath` and the `runProviderCliCommand` invocation,
where spawn failures currently reject around line 1339). This is the correction
from the design review: the named `CONSENSUS_PROVIDER_CLI_MISSING` helper today
lives **only** in refine's preflight (`src/consensus/refine/consensus-refine.ts:2086`),
so centralizing "at the refine path" would leave `evaluate`/`decide`/`plan`/`create`
with non-actionable failures.

**Behavior:** when resolution finds no usable CLI (no explicit arg, no
`CONSENSUS_CLI_PATH`, no plugin-relative file, no `~/.consensus/consensus.mjs`),
the shared surface raises a single actionable `CONSENSUS_PROVIDER_CLI_MISSING`
error stating the shared scripts aren't available and offering two recovery
options:

1. Install the consensus **plugin** (claude/cursor/codex marketplace), or
2. Run the pinned **`install.sh`** one-liner — see the repo README's
   alternative-install section.

**Design decisions:**

- **Single shared surface, all five inherit it.** Define the detection +
  actionable message once in the core resolution/invocation path; have refine's
  existing preflight delegate to (or be unified with) that shared helper rather
  than carrying its own copy. This is what makes the Overview's "added once,
  inherited by the whole family" claim true.
- **Message references the repo README + pinned `install.sh`** (canonical,
  in-repo) rather than a per-skill bundled copy — one documented source of truth.
- **Tests must exercise the missing-CLI path for all five wrappers**, not refine
  alone (see Testing Strategy).

### install.sh (alternative install — pinned fetch)

**Location:** repo root (or a documented path), referenced by README and by the
runtime error message.

**Purpose:** Provision the shared `consensus.mjs` into `~/.consensus/` so a
standalone-installed skill can resolve it — the same shared-script outcome as a
plugin install, via a lighter path that requires neither the full plugin nor a
manual repo clone.

**Acquisition model (decided — pinned fetch from repo):**

- The README and the runtime error surface a one-liner that fetches `install.sh`
  from a **pinned repo ref** (a release tag, not a moving branch), e.g.
  `curl -fsSL https://raw.githubusercontent.com/tkstang/skills/<tag>/install.sh | bash`.
- `install.sh` then fetches the matching pinned `consensus.mjs` from the **same
  ref** and writes it to `~/.consensus/consensus.mjs`.
- Pinning installer + shared script to one ref keeps them version-aligned and
  avoids a moving-target fetch; the README documents the current ref.

**Responsibilities:**

- Fetch the pinned `consensus.mjs` from the repo ref → `~/.consensus/`. When run
  from inside a checkout, prefer the in-tree
  `plugins/consensus/scripts/consensus.mjs` (no network needed).
- Be idempotent (safe to re-run) and Node-22-aware; fail clearly if the fetch or
  write cannot complete.

**Contract consistency:** the resolver fallback (`~/.consensus/consensus.mjs`),
the runtime error message, the README section, and the `install.sh` tests must
all reference this exact acquisition contract and pinned ref.

### README alternative-install section

**Purpose:** Document the pinned `install.sh` one-liner (and the current ref) so
the runtime error can point to it. Single source of truth for "how to make a
standalone consensus skill work" without the full plugin.

### Upstream handoff prompt (category 3)

**Purpose:** A self-contained prompt an agent runs against `open-agent-toolkit`
to add `metadata.internal: true` to the OAT tooling skill definitions at the pack
source, so the flag flows down through `oat init` / `oat tools install` /
`oat sync` into `.agents/skills/**` and survives every sync.

**Why not in this repo:** `.agents/skills/**` are regenerated mirrors; a
hand-edited flag here is overwritten on the next sync. The fix must live upstream.

**This project's deliverable vs. deferred outcome (design-review C1):**

- **Deliverable (in scope — closes this project):** the handoff prompt artifact,
  committed in-repo, naming the target repo (`open-agent-toolkit`), the flag, the
  affected skill set, and the post-sync verification the upstream change should
  produce.
- **Deferred outcome (outside this project's commit boundary):** the actual
  hiding — flag surviving `oat sync` and `.agents/skills/**` dropping from
  `npx skills … --list` (reappearing under `INSTALL_INTERNAL_SKILLS=1`) — is
  **explicitly deferred** to a post-upstream verification, because it depends on
  the upstream change landing and being synced back into this repo. Recorded as a
  follow-up against backlog `BL-260621`, **not** a closure gate for this project.
  Discovery success criteria are updated to this split so a plan cannot "complete
  the prompt" while implying the hiding outcome was already verified.

## Error Handling

- **Missing shared CLI (standalone install):** detected in the **shared core**
  resolution/invocation path; surfaced as the actionable two-option message
  above. Replaces the current cryptic `CONSENSUS_PROVIDER_CLI_MISSING` and is
  emitted for **all five** consensus skills, not refine alone.
- **`install.sh` fetch / re-run:** idempotent write to `~/.consensus/`; safe to
  re-run; fails clearly if the pinned fetch or write cannot complete.
- **Plugin install path unchanged:** existing plugin-relative resolution must
  keep working — the fallback only triggers when that path is absent.

## Testing Strategy

### Unit Tests

- **`resolveConsensusCliPath` resolution order** — explicit arg and
  `CONSENSUS_CLI_PATH` still win; plugin-relative path used when present;
  `~/.consensus/consensus.mjs` used when plugin path absent but shared exists;
  none-found triggers the missing-CLI path. Use temp dirs + a stubbed home; scrub
  `GIT_DIR`/home env so tests don't touch the real `~/.consensus`.
- **Actionable missing-CLI error (all five wrappers)** — assert each of the five
  skill wrappers (refine, evaluate, decide, plan, create) surfaces the shared
  actionable message naming both recovery options + the README/pinned-install
  pointer; assert refine's preflight delegates to the shared helper rather than
  carrying a separate copy (design-review I1).

### Integration Tests

- **Build parity** — `pnpm run build:check` confirms generated `.mjs` matches the
  changed canonical TS; `tests/tooling/generated-output-sync.test.ts` stays green.
- **`install.sh` (pinned fetch)** — with a sandboxed HOME and a mocked pinned
  fetch source, running it writes `~/.consensus/consensus.mjs` from the pinned
  ref; a subsequent resolve finds it; idempotent on re-run; clear failure when
  the fetch/write can't complete. In-checkout mode copies the in-tree script
  without network.

### End-to-End / Manual Verification

- **CLI discovery (the controlled surface):** `npx skills add tkstang/skills --list`
  enumerates expected entries today. The `.agents/skills/**` drop-out (and
  reappearance under `INSTALL_INTERNAL_SKILLS=1`) is a **deferred post-upstream
  verification** (cat-3, design-review C1): it runs only after the
  `open-agent-toolkit` flag lands and syncs into this repo, tracked as a
  follow-up against `BL-260621` — not a gate for this project's completion.
- **Consensus standalone recovery:** simulate a standalone skill dir (no plugin
  tree), confirm the actionable error fires, run `install.sh`, confirm the skill
  then resolves the shared CLI.
- **Plugin discoverability intact:** consensus still presents as a *plugin* via
  plugin-manifest discovery (no `internal` added to consensus).
- **Standalone entries (cat 1):** `session-observer` /
  `export-session-transcript` resolve and run as the only individual entries.
- **skills.sh:** confirm crawl-vs-submission and whether the hosted crawler
  honors `internal`; record the result before any public-listing claim.

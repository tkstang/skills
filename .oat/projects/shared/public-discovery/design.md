# Design: public-discovery

## Overview

This project controls which `SKILL.md` files in the repo are presented as
individually installable skills, across three skill categories that each need a
different treatment. The unifying constraint: we control the `npx skills` CLI
discovery path (it reads our frontmatter and honors `metadata.internal: true`)
but not the skills.sh hosted platform, so the design applies the lever we control
per category and treats skills.sh behavior as something to *verify*, not assume.

The three categories resolve to three independent workstreams. **Standalone
skills** (`session-observer`, `export-session-transcript`) already are the
intended public entries — this category is verification-only. **OAT tooling
skills** (`.agents/skills/**`) are regenerated by the external `oat sync` from
the `open-agent-toolkit` source, so the `internal` flag cannot live in this repo;
the deliverable is an upstream handoff prompt. **Consensus plugin skills**
(`refine`, `evaluate`, `decide`, `plan`, `create`) stay discoverable but are made
*recoverable* when installed standalone: a centralized resolver gains a shared
fallback location, the missing-CLI error becomes actionable, and a repo
`install.sh` provisions the shared script so a standalone-installed skill can be
made to work without the full plugin.

The consensus work is the only code change in this repo, and its blast radius is
small: all five skills resolve the shared CLI through a single function
(`resolveConsensusCliPath`), so the fallback and the actionable error are added
once in canonical TypeScript and inherited by the whole family.

## Architecture

### System Context

Two discovery surfaces and one runtime resolution path interact:

- **`npx skills` CLI** crawls a fixed list of container dirs in the repo
  (`skills/`, `.agents/skills/`, plus plugin-manifest discovery for
  `plugins/consensus`). It honors `metadata.internal: true` (hidden unless
  `INSTALL_INTERNAL_SKILLS=1`). This is the surface we control.
- **skills.sh hosted index** — Vercel-controlled; crawl-vs-submission and whether
  it honors `internal` are unverified. The repo is not yet indexed.
- **Consensus runtime resolution** — at run time each consensus skill locates the
  shared `consensus.mjs` CLI via `resolveConsensusCliPath`. A plugin install
  preserves the whole plugin tree so the existing plugin-relative path resolves;
  a standalone install has only the skill dir, so resolution fails today.

**Key Components:**

- **Category 1 — Standalone skills:** `skills/session-observer`,
  `skills/export-session-transcript`. Self-contained; intended public entries.
  Treatment: **verify only.**
- **Category 2 — Consensus skills:** `plugins/consensus/skills/{refine,evaluate,decide,plan,create}`,
  canonical runtime in `src/consensus/**`, shared CLI at
  `plugins/consensus/scripts/consensus.mjs`. Treatment: **stay discoverable +
  recoverable standalone** (resolver fallback, actionable error, `install.sh`).
- **Category 3 — OAT tooling skills:** `.agents/skills/**` (~56), regenerated by
  external `oat sync`/`oat tools install` from `open-agent-toolkit`. Treatment:
  **upstream handoff prompt** to add `metadata.internal: true` at the pack source.

### Data Flow

Standalone consensus skill run, post-change:

1. User installs one skill via `npx skills add tkstang/skills@refine` (skill dir only).
2. Skill invokes `node ./scripts/consensus-refine.mjs …`, which calls
   `resolveConsensusCliPath`.
3. Resolver checks: explicit arg → `CONSENSUS_CLI_PATH` → plugin-relative path
   (absent) → `~/.consensus/consensus.mjs` (absent on first run) → none found.
4. Runtime emits the actionable missing-CLI message: install the consensus
   plugin, **or** run `install.sh` (see README link).
5. User runs `install.sh`; it provisions `consensus.mjs` into `~/.consensus/`.
6. Next run: resolver finds `~/.consensus/consensus.mjs` and proceeds normally.

## Component Design

### resolveConsensusCliPath (shared-space fallback)

**Location:** `src/consensus/core/consensus-loop.ts` (canonical TS) → built to
the generated `.mjs` runtimes via `pnpm run build`.

**Purpose:** Resolve the shared `consensus.mjs` CLI for all five skills.

**Change:** Insert a shared-space lookup into the existing resolution order, after
the plugin-relative default and before declaring the CLI missing:

```
explicit consensusCliPath
  → env.CONSENSUS_CLI_PATH
  → plugin-relative ../../../scripts/consensus.mjs   (if it exists)
  → ~/.consensus/consensus.mjs                        (NEW; if it exists)
  → (none) → caller surfaces actionable missing-CLI error
```

**Design decisions:**

- **Existence-aware resolution.** Today the function returns the plugin-relative
  path unconditionally (string only). To add a fallback that's only used when the
  plugin path is absent, resolution must check existence and fall through. Keep
  `CONSENSUS_CLI_PATH` and explicit arg as highest-priority overrides (unchanged).
- **`~/.consensus/` shared space** — fixed literal `path.join(os.homedir(), '.consensus', 'consensus.mjs')`.
  Chosen over XDG (`~/.local/share/consensus/`) because (a) the toolchain already
  uses home-root dotdirs (`~/.agents`, `~/.claude`, `~/.cursor`, `~/.codex`) and
  (b) a single literal path can't drift between the resolver and `install.sh` (no
  `$XDG_DATA_HOME` branching to keep in sync) — drift is the silent-breakage risk.
- **Dependency-free / Node stdlib only** (`os`, `path`, `fs`) per repo runtime
  contract.

### Actionable missing-CLI error

**Location:** the `CONSENSUS_PROVIDER_CLI_MISSING` path
(`src/consensus/refine/consensus-refine.ts:2090` and equivalent surfaces across
the family) — centralize the message so all five skills emit the same guidance.

**Behavior:** when no CLI resolves, fail with a message that states the shared
scripts aren't available and offers the two recovery options:

1. Install the consensus **plugin** (claude/cursor/codex marketplace), or
2. Run **`install.sh`** — see the repo README's alternative-install section.

**Design decision:** the message references the **repo README + `install.sh`**
(canonical, in-repo) rather than a hosted curl one-liner or per-skill bundled
copy. Nothing extra to ship per skill; one documented source of truth.

### install.sh (alternative install)

**Location:** repo root (or a documented path), referenced by README.

**Purpose:** Provision the shared `consensus.mjs` into `~/.consensus/` so a
standalone-installed skill can resolve it — the same shared-script outcome as a
plugin install, via a different path.

**Responsibilities:**

- Resolve/obtain the shared `consensus.mjs` (from the repo checkout when run
  there; otherwise fetch the canonical script) and write it to `~/.consensus/`.
- Be idempotent (safe to re-run) and Node-22-aware.

**Open design detail (resolve during implementation):** exact acquisition of
`consensus.mjs` when run outside a full checkout (copy from cwd vs. fetch). When
run from a repo clone, copying the built `plugins/consensus/scripts/consensus.mjs`
is sufficient; the fetch path is the only nuance.

### README alternative-install section

**Purpose:** Document the `install.sh` path so the runtime error can point to it.
Single source of truth for "how to make a standalone consensus skill work."

### Upstream handoff prompt (category 3)

**Purpose:** A self-contained prompt an agent runs against `open-agent-toolkit`
to add `metadata.internal: true` to the OAT tooling skill definitions at the pack
source, so the flag flows down through `oat init` / `oat tools install` /
`oat sync` into `.agents/skills/**` and survives every sync.

**Why not in this repo:** `.agents/skills/**` are regenerated mirrors; a
hand-edited flag here is overwritten on the next sync. The fix must live upstream.

**Deliverable:** the prompt artifact (committed in-repo, e.g., under the project
or a docs/handoff location), naming the target repo, the flag, the affected skill
set, and the verification (`npx skills … --list` drops them;
`INSTALL_INTERNAL_SKILLS=1 … --list` shows them).

## Error Handling

- **Missing shared CLI (standalone install):** detected in
  `resolveConsensusCliPath`'s fallthrough; surfaced as the actionable two-option
  message above. Replaces the current cryptic `CONSENSUS_PROVIDER_CLI_MISSING`.
- **`install.sh` re-run / partial state:** idempotent write to `~/.consensus/`;
  safe to re-run.
- **Plugin install path unchanged:** existing plugin-relative resolution must
  keep working — the fallback only triggers when that path is absent.

## Testing Strategy

### Unit Tests

- **`resolveConsensusCliPath` resolution order** — explicit arg and
  `CONSENSUS_CLI_PATH` still win; plugin-relative path used when present;
  `~/.consensus/consensus.mjs` used when plugin path absent but shared exists;
  none-found triggers the missing-CLI path. Use temp dirs + a stubbed home; scrub
  `GIT_DIR`/home env so tests don't touch the real `~/.consensus`.
- **Actionable error content** — asserts the message names both recovery options
  and the README pointer; asserts all five skills route through the same message.

### Integration Tests

- **Build parity** — `pnpm run build:check` confirms generated `.mjs` matches the
  changed canonical TS; `tests/tooling/generated-output-sync.test.ts` stays green.
- **`install.sh`** — running it provisions `~/.consensus/consensus.mjs`
  (sandboxed HOME); a subsequent resolve finds it; idempotent on re-run.

### End-to-End / Manual Verification

- **CLI discovery (the controlled surface):** `npx skills add tkstang/skills --list`
  enumerates expected entries; after the upstream cat-3 change lands,
  `.agents/skills/**` drop out and reappear under `INSTALL_INTERNAL_SKILLS=1`.
- **Consensus standalone recovery:** simulate a standalone skill dir (no plugin
  tree), confirm the actionable error fires, run `install.sh`, confirm the skill
  then resolves the shared CLI.
- **Plugin discoverability intact:** consensus still presents as a *plugin* via
  plugin-manifest discovery (no `internal` added to consensus).
- **Standalone entries (cat 1):** `session-observer` /
  `export-session-transcript` resolve and run as the only individual entries.
- **skills.sh:** confirm crawl-vs-submission and whether the hosted crawler
  honors `internal`; record the result before any public-listing claim.

## Open Questions

- **install.sh acquisition path** when run outside a full checkout (copy vs.
  fetch the canonical `consensus.mjs`) — settle in implementation.
- **skills.sh crawler semantics** — auto-crawl vs submission-gated; honors
  `internal`? Verification task, not a code decision.

## Clarifying Questions

### Question 1: Do we actually control skill discovery?

**Q:** "We don't really control discovery [of the skills], do we?"
**A (established together):** We control it on the path that matters and not on
the one we don't:

- **CLI path â€” we control it.** `npx skills add tkstang/skills --list` runs the
  `vercel-labs/skills` CLI against our repo and reads our `SKILL.md` frontmatter.
  That CLI **already honors `metadata.internal: true`** (documented): an internal
  skill drops out of normal discovery and only reappears with
  `INSTALL_INTERNAL_SKILLS=1`. This is the real, in-our-control lever and it is
  the primary install surface.
- **skills.sh hosted platform â€” we do NOT control it.** Its crawler, ranking,
  and inclusion rules are Vercel's. Unverified whether it auto-crawls public
  repos or is submission-gated, and unverified whether the hosted crawler
  respects `internal` the same way the CLI does.

**Decision:** Scope the project to the lever we control **plus** verification of
the part we don't, rather than assuming platform control we lack.

### Question 2: How should we scope the project?

**Q:** Lever + verification / lever only / verify-first?
**A:** **Lever + verification** (full backlog scope). Apply `metadata.internal`
to the skills that should not be standalone-installable (surviving `oat sync`)
AND verify/decide the skills.sh crawl-vs-submission strategy.

## Solution Space

The backlog already did the divergent exploration (see its Findings). The
chosen direction below is the convergence; it is recorded here, not re-derived.

### Chosen Direction

**Approach:** Use `metadata.internal: true` as the single exclude lever on the
CLI discovery path, applied per category, and verify the skills.sh hosted
behavior empirically before any public-listing claim.

**Rationale:** `internal` is the only mechanism the `vercel-labs/skills` CLI
documents and honors. `skills.sh.json` is display-only (grouping/order, not
discovery). There is no `.skillsignore`/exclude flag. Default CLI discovery
already walks a fixed list of container dirs â€” including `.agents/skills/` â€” so a
repo-root `SKILL.md` does NOT suppress anything (the earlier "root SKILL.md
lever" hypothesis was disproven 2026-06-20).

**User validated:** Yes â€” scope confirmed as "Lever + verification".

## Key Decisions

1. **Exclude mechanism:** `metadata.internal: true` in `SKILL.md` frontmatter is
   the lever. Verify with `npx skills add tkstang/skills --list` (internal skills
   drop out) vs `INSTALL_INTERNAL_SKILLS=1 â€¦ --list` (they reappear).
2. **Category 3 (OAT tooling `.agents/skills/**`):** must end up `internal`. The
   committed files are **regular files regenerated by the external `oat sync`
   CLI** (verified: git mode `100644`, not symlinks; `.agents/` is not
   gitignored). A hand-edited flag is clobbered on the next sync â€” the flag must
   come from the sync source or a deterministic post-sync step. **Where the flag
   lives is the primary open design question.**
3. **Category 2 (consensus skills â€” all five: `refine`, `evaluate`, `decide`,
   `plan`, `create`):** keep discoverable, not hidden â€” leave them visible so the
   plugin stays marketable, but make a standalone install **recoverable** rather
   than cryptically broken. The shared-core resolver gains a `~/.consensus/`
   fallback, the missing-CLI failure becomes an actionable message with two
   recovery options (install the consensus plugin, or run the pinned
   `install.sh`), and a repo `install.sh` (pinned fetch from a repo ref)
   provisions the shared script. Scope is all five current consensus skills, not
   only the originally-named `refine`/`evaluate` (mechanism detailed in
   `design.md`).
4. **Category 1 (standalone skills):** confirm `session-observer` and
   `export-session-transcript` remain the only individually-installable entries
   and resolve/run correctly.
5. **skills.sh strategy:** confirm auto-crawl vs submission-gated and choose the
   listing strategy accordingly; record the verification result before any
   public-listing claim.

## Constraints

- **Do not hand-edit `oat sync`-generated files** (`.agents/skills/**`); the flag
  must survive a sync. `oat` is an external CLI, not vendored.
- **Consensus SKILL.md/runtime changes are shipped-skill content/behavior
  changes** â†’ require a skill **version bump** with top-level `version` and
  `metadata.version` kept in sync (`scripts/validate.mjs`), enforced by
  `scripts/validate-skill-versions.mjs`. This applies to whichever of the five
  consensus skills change (`refine`/`evaluate` are at `0.1.1`; `decide`/`plan`/
  `create` bump from their current versions).
- **Consensus runtime behavior changes go through canonical TS under `src/`**,
  then `pnpm run build` regenerates the `.mjs`; never hand-edit generated
  `// GENERATED` outputs.
- Runtime/shipped skill code stays dependency-free (Node stdlib only).
- Verification must use the real CLI path, per repo convention not to claim
  provider/marketplace/skills.sh discovery as complete until the live path is
  verified.

## Findings

### Critical

- **Shipped wake envelopes bypass automatic-control classification and loop suppression** (`src/transcript/core/runtimes.ts:122`)
  - Issue: `parseAutomaticControlEnvelope` accepts only a JSON object with camel-case `leaseId`, `pinnedPeer`, and `range`, while both shipped adapters emit XML (`codex-stop.mjs:66-74`, `cursor-stop.mjs:54-62`) with `lease_id`, `peer`, and `records`. A direct probe using the real `wakeEnvelope()` output produced an ordinary `{ role: "user", kind: "message" }` entry with no `displayRole`, `origin`, or `automaticControl`; feeding that turn plus `Acknowledged.` to `selectCompletedContinuation` returned `continuation: true` and `budgetCost: 1`. The production control message is consequently rendered as `User`, counts as genuine user engagement, can be mistaken for human direction/authorization, and can recursively wake the other agent on an acknowledgment. The existing fixtures use a different JSON envelope, so all focused tests pass without exercising the actual adapter-to-normalizer contract.
  - Fix: Establish one canonical wire format and align both hook emitters, all three runtime normalizers, the renderer/classifier/completion selector, runtime references, and fixtures. Add cross-contract tests that pass the literal output of `wakeEnvelope()` and `cursorWakeEnvelope()` through Claude/Codex/Cursor normalization and assert `Hook/control (automatic)`, non-human engagement, and zero continuation budget for an automatic acknowledgment/replay.
  - Requirement: Prompt Part 1.8; Collaboration Protocol authority and loop-prevention rows; Shared continuation invariants.

### Important

- **Runtime-reference routing uses the peer runtime instead of the acting harness** (`skills/session-observer-collab/SKILL.md:74`)
  - Issue: The skill and public guide direct the agent to resolve the *peer* runtime and load that matching reference. The references themselves are explicitly acting-runtime setup contracts (`runtime-codex.md` installs/configures the local Codex Stop hook; `runtime-cursor.md` consumes the local Cursor Stop event; `runtime-claude-code.md` probes the local Monitor). In a normal cross-runtime pair, a local Codex agent observing Claude is therefore routed to the Claude Monitor recipe instead of its Codex lifecycle adapter, and the local Claude agent is routed toward Codex setup. This contradicts the design's per-harness capability probe and can leave both sides at the wrong wake tier.
  - Fix: Route the one loaded reference by the acting/self runtime established by `whoami`; use the peer runtime only for the observation pin and peer transcript. Update the skill, user guide, and routing test with cross-runtime examples such as acting Codex → peer Claude and acting Claude → peer Codex.
  - Requirement: Prompt Part 2, Section 2; Design Runtime references and capability selection.

### Medium

- **Transcript-global content deduplication drops repeated queued human messages** (`src/transcript/core/runtimes.ts:650`)
  - Issue: `attachmentOnlyContents` deduplicates `queued_command` attachments by prompt text across the entire transcript, and `queuedOperationContents` suppresses every matching attachment regardless of which queue transaction produced it. Two distinct attachment-only queued messages with identical text normalize to one entry in a direct probe. Repeating the same approval/request later is legitimate human input and must not be silently removed merely because its text matches an earlier queue item.
  - Fix: Correlate an attachment only with the enqueue/remove transaction or delivery copy it duplicates; preserve separate occurrences with identical content. Add fixtures for two distinct identical attachment-only messages and for an old enqueue followed by a later independent identical attachment.
  - Requirement: Prompt Part 1.1 queued-input rendering/deduplication.

- **Conflicting harness identity signals fail without the required candidate list** (`src/transcript/session-observer/lib/observe.ts:92`)
  - Issue: When more than one harness signal is present, `harnessIdentity` returns `null`; `resolveSelfIdentity` then returns `noMatch` without discovering or listing the plausible sessions. The committed test explicitly expects exit 2 / `noIdentity` for simultaneous Claude and Codex signals, but the authoritative acceptance text requires ambiguous environments to exit non-zero *with a candidate list*. This leaves the arming handshake fail-closed but not recoverable from the command's output.
  - Fix: Preserve conflicting runtime/session signals as an ambiguity result, discover candidates for each signaled runtime in the requested cwd, and return them through the CLI ambiguity payload (exit 3). Keep the no-guess behavior and add coverage for conflicting runtime indicators and conflicting session-ID aliases.
  - Requirement: Prompt Part 1.3 and Base `session-observer` Self identity acceptance row.

### Important

- **SKILL.md documents `state reset --session` but the CLI does not implement it** (`.agents/skills/session-observer/SKILL.md:56`)
  - Issue: The Subcommands table lists a row `state reset --session <r:id>` — "Reset one session — Zeroes `lastRecordIndex`". The CLI's `runState` `reset` case (`scripts/session-observer.mjs:544-558`) never reads `args.session` and never calls `stateLib.resetBySession`; it requires `--runtime` (errors with exit 1 if absent or `auto`) and unconditionally calls `resetByRuntime(runtime)`. So `state reset --session codex:abc` exits 1 ("--runtime is required for state reset"), and `state reset --runtime codex --session codex:abc` silently resets *every* codex session, not just the named one. The `resetBySession` function exists and is exported from `state.mjs:258`, but no CLI path reaches it. SKILL.md tells the agent a per-session reset is available when it is not — an agent following the doc will either hit a confusing exit-1 error or silently wipe more state than intended.
  - Fix: Pick one and apply it consistently:
    - **(preferred)** Add a follow-up implementation task to wire `--session` into the `reset` case: when `args.session` is set, split `runtime:id` and call `stateLib.resetBySession(runtime, id)` instead of `resetByRuntime`; relax the `--runtime` requirement when `--session` is supplied. This matches spec Section 6 ("`state reset --session <r>:<id>` — Reset one entry") and keeps the doc honest.
    - **(doc-only alternative)** If per-session reset is intentionally deferred from v1, remove the `state reset --session <r:id>` row from the SKILL.md Subcommands table and note it as a deferred capability. Spec Section 6 would then also need a deviation note.
  - Note: this is a p05-surfaced gap whose root cause is in p04 CLI code; either fix is acceptable for closing the p05 review, but the artifacts and code must agree.
  - Requirement: spec §6 `state` subcommand operations table.

  - Issue: The Troubleshooting heading reads "Lock exhausted (exit 1, `EBUSY`)". `state.mjs`'s lock-retry exhaustion throws a plain `Error` and the CLI surfaces it via `emitError(... , 1)` as `[session-observer] Failed to ...` text — there is no `EBUSY` code on stdout/stderr or in any JSON payload. The `EBUSY` label comes from the spec's failure-mode table (#9) as a conceptual category, not an observable string. An agent grepping output for `EBUSY` will not find it.
  - Suggestion: Change the heading to "Lock exhausted (exit 1)" and describe the symptom as "the CLI exits 1 with a `Failed to ...` message after retrying the state lock"; keep the `rm ...state.json.lock` recovery as-is.

  - Suggestion: Use the skill-relative form `<skill-dir>/scripts/lib/runtimes.mjs` (or just `runtimes.mjs` unqualified, as line 159 already does) for consistency with SKILL.md and the design decision. Low priority — purely a consistency nit in a reference doc.

### Medium

- **`load()` / `readState()` perform writes outside the lock** (`scripts/lib/state.mjs:93-114`, `181-184`)
  - Issue: `load()` is documented as the read-only path and deliberately does not
    acquire `state.json.lock`. But the function it calls, `readState()`, writes
    files as a side effect: a corrupt-state backup (`writeFile`, line 109) and a
    v0 migration backup (`migrateIfNeeded`, line 125). If `load()` runs concurrently
    with a `mutate()` (which is lock-protected), or two `load()` calls race on a
    corrupt file, two writers can both create backups / both observe the corrupt
    file. The design (`design.md:271-272`) and spec (`spec §state` line 293) place
    corrupt-backup and migration inside the locked write protocol ("Read current
    `state.json` ... preserve the corrupt file as `state.json.corrupt-...bak`" is
    step 3 of the locked sequence). The current split means the side-effecting read
    path is unsynchronized.
  - Fix: Either (a) acquire the lock in `load()` before calling `readState()` when
    a backup might be written, or (b) keep `load()` strictly side-effect-free —
    return empty state on corrupt/old schema without writing a backup — and let the
    next `mutate()` (which is locked) perform the backup. Option (b) keeps `load()`
    honest to its "read-only" docstring and matches the spec's "backup happens in
    the locked write protocol" wording.

- **Migration is not persisted when only `load()` is called** (`scripts/lib/state.mjs:119-131`)
  - Issue: Plan p01-t02 step 1 says `migrateIfNeeded` should write the `.bak`
    "then upgrades in place." `migrateIfNeeded` returns an upgraded in-memory
    object but never writes the upgraded `state.json` back to disk. Test 9 only
    asserts the `.bak` exists and the *returned* object has `schemaVersion: 1`; it
    never re-reads `state.json` from disk. So after a `load()` on a v0 file, the
    on-disk `state.json` is still v0 and the `.bak` already exists — a subsequent
    `load()` will re-trigger migration and overwrite `state.v0.json.bak` with the
    (still v0) file again. The disk file is only actually upgraded if a `mutate()`
    happens to run later. "Upgrades in place" is therefore only partially true.
  - Fix: Persist the migrated state. Cleanest: route migration through the locked
    write path so the upgraded `state.json` is written atomically once, and guard
    the `.bak` write so it is not overwritten if it already exists (e.g.
    `open(bakPath, 'wx')` or skip when `access(bakPath)` succeeds). Add a test
    assertion that re-reads `state.json` from disk and confirms `schemaVersion: 1`
    was persisted.

- **Backup writes are non-atomic and can clobber** (`scripts/lib/state.mjs:109`, `125`)
  - Issue: Corrupt and v0 backups use a plain `writeFile`, not the temp+fsync+rename
    protocol the module otherwise enforces for `state.json`. The v0 backup path
    `state.v0.json.bak` is a fixed name, so a second migration attempt silently
    overwrites a previously captured backup (losing the original v0 data the backup
    was meant to preserve). The corrupt backup uses a `Date.now()` suffix so it is
    collision-resistant, but is still a non-atomic write of potentially important
    recovery data.
  - Fix: For the v0 backup, refuse to overwrite an existing `.bak` (open with `'wx'`
    or check `access` first). Consider writing backups via temp+rename for crash
    safety, consistent with the rest of the module.

    becomes truly side-effect-free) or update it to state that corrupt/old-schema
    files trigger a backup write.

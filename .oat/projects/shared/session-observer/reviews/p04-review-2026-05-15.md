### Minor

- **`catch-up` advances state even when `newRecords === 0`** (`scripts/session-observer.mjs:418-427`)
  - Issue: The spec's end-to-end flow (Section 7, step 11) says `if digest.range.newRecords === 0: emit "no new records" header; exit 0` — i.e. step 12 (`state.markRead`) is reached only when there are new records. `runCatchUp` unconditionally calls `stateLib.markRead` after building the digest, with no `newRecords === 0` short-circuit and no dedicated "no new records" header. Behavior is still correct in practice (a zero-delta `markRead` re-writes the same `lastRecordIndex`, so the offset does not regress, and `formatHeader` prints `**new records:** 0`), but it diverges from the spec's described control flow and incurs a redundant locked state write on every no-op `catch-up`.
  - Suggestion: After `buildDigest`, branch on `digest.range.newRecords === 0`: emit a concise "no new records" digest/header and `return` before `markRead`. This matches the spec flow and the integration test's intent ("second reports 'no new records'"). Confirm with the final reviewer whether the redundant write is acceptable or should be guarded.

- **`tierOf` uses raw string equality for cwd comparison** (`scripts/lib/rank.mjs:47-54`)
  - Issue: `tierOf` compares `recordedCwd` and `targetCwd` with `===` and `startsWith`, not `realpath`-normalized paths. Spec failure mode #14 ("Symlinked cwd — `realpath` both sides before comparison") is therefore not satisfied. This is pre-existing from p03 (already recorded as a p03 Minor) and surfaces here only because the CLI is the first consumer that feeds real `--cwd` values into `rank`.
  - Suggestion: Carry the existing p03 finding to the final review; normalize both sides via a `realpathSafe` helper before tier classification. Out of strict p04 scope — no change required for the p04 gate.

- **`renderJson` does not validate or stamp a hard `schemaVersion` failure path** (`scripts/lib/digest.mjs:309-311`)
  - Issue: `renderJson` is a thin `JSON.stringify`. The CLI never emits exit code 4 (schema mismatch) anywhere — the exit-code contract documents `4 = schema mismatch`, but no code path produces it. This is acceptable for v1 (no schema-versioned input is parsed by the CLI; `state.mjs` handles its own migration), but the unused exit code is worth a one-line note so the final reviewer confirms it is intentionally reserved rather than missing.
  - Suggestion: No code change. Confirm at final review that exit 4 is intentionally reserved for a future schema-mismatch surface (consistent with `state.mjs` migration being non-fatal/exit-0).

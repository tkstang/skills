### Minor

- **Codex cwd cache provides no actual speedup — transcript is re-parsed every call** (`.agents/skills/session-observer/scripts/lib/locate.mjs:303-311`)
  - Issue: `discoverCodex` consults the cwd cache to skip the `extractMeta` call
    for `recordedCwd` (lines 284-301), but immediately afterward it calls
    `extractMeta('codex', transcriptPath)` again unconditionally (line 307) just
    to obtain `sessionId`. `extractMeta` runs `readRecords`, which reads and
    fully parses the whole transcript. So on a cache hit the file is still read
    and parsed in full — the cache saves nothing. This defeats the design's
    stated rationale (`design.md:176`: "avoids re-parsing Codex transcripts for
    cwd extraction on every poll/check") and the spec's caching intent
    (`spec §4`, codex row). It is purely a performance/intent gap — discovery
    results are still correct.
  - Why the test still passes: `locate.test.mjs:175-227` rewrites the transcript
    with a different `cwd` but keeps the same `sessionId` (`codex-sess-001`), so
    the unconditional re-parse coincidentally yields a `sessionId` consistent
    with the cached `recordedCwd`. The test proves the cwd *value* is cached but
    does not detect that the file is re-parsed anyway.
  - Fix: extend the cache entry to also carry `sessionId`
    (`cache[key] = { recordedCwd, sessionId }`), parse once on a cache miss,
    reuse `meta` for both fields, and on a cache hit take both values from the
    cache with no second `extractMeta` call. Then strengthen the cache test to
    also rewrite the `sessionId` in the transcript content and assert the
    *cached* `sessionId` is returned on the second `discover` — which would fail
    against the current implementation, proving the cache is genuinely consulted.

- **Tier matching uses plain string equality / prefix, not realpath-normalized comparison** (`.agents/skills/session-observer/scripts/lib/rank.mjs:47-54`)
  - Issue: `tierOf` compares `recordedCwd === targetCwd` and
    `recordedCwd.startsWith(targetCwd + '/')` as raw strings. The spec defines
    Tier A as `realpath(recordedCwd) === realpath(targetCwd)` (`spec §4`,
    line 153) and failure mode #14 (`spec §8`) requires `realpath` on both
    sides for symlinked cwds; `rank.mjs`'s own header comment block and
    `rank.test.mjs:192` ("realpathSafe handles ENOENT without throwing") plus
    the spec helper list (`spec §7`: `realpathSafe(p)`) all anticipate a
    `realpathSafe` helper that is not present. In practice this is benign for
    p03: the Claude Code direct-lookup path sets `recordedCwd = targetCwd` from
    the *same string* the CLI will pass to `rank`, so exact matches hold; Codex
    cwds and the CLI's `--cwd` would only diverge under symlinks.
  - Fix (defer to p04/final): either add a `realpathSafe(p)` helper and
    normalize both sides inside `tierOf`, or have the CLI resolve `targetCwd`
    and each `recordedCwd` via `realpath` before calling `rank` and document
    that `rank` operates on pre-normalized paths. Update `design.md`'s `rank`
    section to state where realpath normalization lives so design and spec
    agree. The `rank.test.mjs` "realpathSafe" test currently asserts only that
    `rank` does not throw on a nonexistent path — it does not exercise symlink
    resolution.

## Findings

### Critical

- **Tool-call marker uses `[Tool: Name]` instead of spec's `[Name]`** (`.agents/skills/session-observer/scripts/lib/runtimes.mjs:347`, `:424`)
  - Issue: `claudeEntriesFromContent` emits `` `[Tool: ${name}] ${argsStr}` `` and `normalizeCodex` emits `` `[Tool: ${name}] ${argsStr}` ``. The source-of-truth spec (`.superpowers/specs/2026-05-14-session-observer-design.md`) defines `tool_call` as `[ToolName] truncated-args` — section 5 entry-kind table (line 224) and the JSON example (`"[Edit] {\"file_path\":...}"`, line 208). Stoa's reference adapters confirm the contract: `claude-code.ts toolUseFromBlock` returns `` `[${name}] ...` `` and `codex.ts functionCallFromRecord` returns `` `[${name}] ...` `` — no `Tool: ` prefix. Verified empirically: `normalizeEntries` on `claude-code/typical.jsonl` produces `[Tool: Read] {"file_path":...}` and on `codex/typical.jsonl` produces `[Tool: shell] {"command":"ls -la"}`. The plan's p04-t04 integration test (plan.md:549–551) asserts stdout "contains no `[Edit]` / `[Bash]` markers" by default and "contains compact tool markers" with `--include-tools` — those assertions expect bare `[Edit]`/`[Bash]`, which the current output (`[Tool: Edit]`) will not satisfy. `digest.mjs` renders `entry.text` verbatim, so the wrong prefix propagates straight to CLI output.
  - Fix: Change both call sites to emit `` `[${name}] ${argsStr}` `` (drop the `Tool: ` prefix). Update the matching assertion in `tests/session-observer/runtimes.test.mjs` if any test pins the `Tool: ` substring (current tests assert `startsWith('[')` only, so they will still pass — but add a positive assertion that the marker is `[<name>]` to lock the contract).

- **Tool-result marker uses `[Tool → result]` instead of spec's `[ToolName → result]`, and `toolName` field is missing on `tool_result` entries** (`.agents/skills/session-observer/scripts/lib/runtimes.mjs:363`)
  - Issue: `claudeEntriesFromContent` emits `` `[Tool → result] ${resultText}` `` with no tool name interpolated, and the returned entry omits `toolName`. The spec entry-kind table (line 225) specifies `tool_result` as `[ToolName → result] truncated-output`, and the spec JSON example (lines 210–211) shows `"text": "[Edit → result] applied 1 edit..."` with `"toolName": "Edit"` on the `tool_result` entry. `design.md`'s `DigestEntry` model (lines 314–322) declares `toolName?: string` as a shared optional field. Verified empirically: `normalizeEntries` on `claude-code/typical.jsonl` produces `[Tool → result] export function main()...` with `toolName: undefined`.
  - Fix: Resolve the tool name for `tool_result` blocks and emit `` `[${name} → result] ${resultText}` ``, and set `toolName: name` on the entry. Claude Code `tool_result` blocks carry `tool_use_id` (e.g. `tool_1`), not the tool name directly — correlate `tool_use_id` back to the matching `tool_use` block's `name` within the record set (build a `tool_use_id → name` map across records during `normalizeClaudeCode`). If correlation is out of scope for this layer, escalate to a plan delta so `digest.mjs` does not inherit an unnamed result marker; do not silently ship `[Tool → result]`.

### Minor

- **Codex `extractMeta` reads `cwd` only at the record top level, not from `payload`** (`.agents/skills/session-observer/scripts/lib/runtimes.mjs:305`)
  - Issue: The codex branch does `asString(record.cwd)`. The fixtures place `cwd` at the top level of the `session_started` record, so this matches the defined fixture contract and all tests pass. But the spec (section 4, line 135) says cwd is extracted "from the session-meta record" without pinning placement; if a real Codex transcript nests `cwd` inside `payload`, extraction would silently return `null` → Tier C. This is contained (it degrades to the documented no-cwd path) and the p06 manual probe will surface it against real transcripts.
  - Suggestion: Add a `payload.cwd` fallback (`asString(record.cwd) ?? asString(payload.cwd)`) for robustness, or note the assumption explicitly in `references/transcript-formats.md` (p05-t03).

- **`stringifyArgs` truncation can split a multi-byte/JSON token mid-string** (`.agents/skills/session-observer/scripts/lib/runtimes.mjs:67`)
  - Issue: `truncate` slices by character count and appends `...`, so a truncated JSON args string is not valid JSON. This matches Stoa's behavior exactly (lossy-by-design marker text, not parsed downstream) and the spec explicitly says args are "ellipsized" — so this is correct. Noting only that downstream code must never `JSON.parse` an entry's `text`.
  - Suggestion: No change required; flagged for p04 awareness.

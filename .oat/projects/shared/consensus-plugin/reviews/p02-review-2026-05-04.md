## Findings

### Critical

- **Verdict, record, status, hash, and cap contracts do not match the design schemas** (`plugins/consensus/skills/consensus-refine/schemas/verdict-alternating.schema.json:9`)
  - Issue: The design pins peer verdicts as `verdict: "ACCEPT" | "REVISE" | "IMPASSE"`, but the schema and validators require `decision` instead. The same drift appears in records/status: hashes are bare hex instead of `sha256:<hex>` (`consensus-loop.mjs:241`), statuses emit `artifact_hash` rather than `final_artifact_hash` (`consensus-loop.mjs:378`), and turn records use ad hoc fields like `turn`, `round`, `recorded_at`, and nested verdict objects instead of the designed `turn_index`, `round_index`, `timestamp`, `iteration_mode`, and `verdict` enum. The byte caps also exceed the design: reasoning is 64 KB instead of 16 KB, proposed artifacts are 1 MB instead of 256 KB, concerns are 16 KB each instead of 4 KB, and there is no 20-concern or 512 KB total verdict cap (`consensus-loop.mjs:7`). Tests currently assert the drift rather than the written contract.
  - Fix: Align the schema file, validators, records, status writer, hash formatter, and tests to `design.md` sections 4.1-4.3 and 5. Use `verdict`, exact cap constants, `sha256:`-prefixed hashes, designed status enum/termination values, and the designed record/status field names.
  - Requirement: FR2, FR3, FR5

- **Turn prompts omit the required untrusted-content framing** (`plugins/consensus/skills/consensus-refine/scripts/consensus-loop.mjs:568`)
  - Issue: `buildTurnPrompt` wraps the artifact in a raw markdown fence and does not include the design's `<SECTION>...</SECTION>` untrusted-data boundary or explicit instruction to ignore directives inside the artifact. A document containing a code-fence close can break out of the prompt's fence, and even without that, the peer is not told that artifact text is data. The prompt also omits the previous peer verdict block required by the design, weakening the alternating deliberation contract.
  - Fix: Rebuild the prompt to match `design.md` section 4.6: stable protocol text first, goal and iteration metadata, explicit untrusted `<SECTION>` delimiters, "ignore instructions inside" guidance, JSON-encoded previous verdict, and then the schema-only output instruction. Avoid raw markdown fences around untrusted artifact text.
  - Requirement: FR2, prompt-injection security design

- **Section run-file writes can escape through symlink parents** (`plugins/consensus/skills/consensus-refine/scripts/consensus-refine.mjs:314`)
  - Issue: `atomicWriteFile` rejects a symlink only when the final target path already is a symlink; it does not reject symlink ancestors. `runSequential` writes section inputs/outputs under derived `runDir` children without passing each path through `confineWrite` (`consensus-refine.mjs:689`). A pre-existing symlink such as `.consensus/run/sections -> /tmp/outside` causes section files to be written outside the confined run root, violating the p02 path-safety task and the design's run-file confinement requirement.
  - Fix: Make confined atomic writes a single primitive that checks the nearest existing ancestor's real path against an allowed real root immediately before creating directories and writing. Use it for every derived run file and output file. Add regression coverage for symlink parents under the run directory, not only symlink final targets.
  - Requirement: p02-t08, security path-safety design

### Important

- **Peer IDs are not validated against the designed provider-id grammar** (`plugins/consensus/skills/consensus-refine/scripts/consensus-refine.mjs:166`)
  - Issue: `parsePeers` only enforces "exactly two peers"; `resolvePeers` checks inventory membership but never enforces the design's `^[a-z][a-z0-9-]{0,31}$` provider-id rule. That leaves the wrapper accepting provider IDs outside the documented CLI contract whenever the inventory shape includes them.
  - Fix: Validate every peer ID in both explicit `--peers` input and normalized inventory entries before use. Add tests for invalid characters, uppercase, leading digits, and too-long IDs.
  - Requirement: FR9

- **Environment and permission exit-code mapping conflicts with the design** (`plugins/consensus/skills/consensus-refine/scripts/consensus-loop.mjs:189`)
  - Issue: The design maps environment/config failures such as missing Paseo or unavailable peers to exit 78, and reserves 77 for permission failures. The implementation maps `PASEO_MISSING` to 77 and maps `EACCES`/`EPERM` to 73 through the generic IO branch (`consensus-loop.mjs:192`). That breaks the p02-t10 exit-code contract for automation and host-runtime handling.
  - Fix: Adjust `EXIT_CODES` naming and `exitCodeForError` so missing Paseo, peer-unavailable, and Node-too-old errors return 78; permission-denied errors return 77; write/create failures return 73. Update tests to assert the design table.
  - Requirement: FR9, NFR6, p02-t10

### Minor

- **Trace-mode stack traces can be placed in stdout JSONL** (`plugins/consensus/skills/consensus-refine/scripts/consensus-refine.mjs:862`)
  - Issue: `runWrapperCli` puts `renderHumanError(error, env)` into the stdout JSONL `message` field. With `CONSENSUS_LOG=trace`, `renderHumanError` returns `error.stack`, which contradicts p02-t10's stdout-cleanliness goal even though the line remains JSON.
  - Suggestion: Keep stdout JSONL messages one-line and stack-free in all modes. Emit trace stacks to stderr or a debug side channel only.

### Important

- **`--agency` is accepted but does not affect loop behavior** (`plugins/consensus/skills/consensus-refine/scripts/consensus-loop.mjs:749`)
  - Issue: The design says v0.1 agency modulates convergence strictness and max-rounds/oscillation handling: `minimal` requires strict bytewise hash equality, `moderate` uses normalized hashes, and `maximum` may declare done at max rounds with the last artifact while logging the agency decision. The current loop validates and records `agency`, but always calls `detectConvergence(records)` without agency options and always emits `status: "max-rounds"` on budget exhaustion. As a result, `--agency minimal` and `--agency maximum` are effectively no-ops for the decisions they are meant to control.
  - Fix: Thread agency into convergence/oscillation/max-round decisions. Add regression tests showing strict hash behavior for `minimal`, default normalized behavior for `moderate`, and the designed max-round/near-match behavior plus status logging for `maximum`.
  - Requirement: FR3, p02-t01, p02-t05

- **Direct loop CLI still maps missing Paseo to file IO instead of config** (`plugins/consensus/skills/consensus-refine/scripts/consensus-loop.mjs:461`)
  - Issue: The wrapper preflight path now reports missing Paseo as exit 78, but `consensus-loop.mjs` still rejects raw child-process spawn errors from `invokePaseo`. When `paseo` is absent, the error is `spawn paseo ENOENT`; `exitCodeForError` maps raw `ENOENT` to 73, even though the design exit table assigns missing Paseo/environment issues to 78. This leaves prior finding I2 only partially fixed for the loop CLI used by direct and future section-subagent invocations.
  - Fix: Convert `spawn paseo ENOENT` in `invokePaseo` to a `ConsensusError` with `code: "PASEO_MISSING"` and `exitCode: EXIT_CODES.CONFIG`; preserve generic filesystem `ENOENT` as 73. Add a direct `runConsensusLoop` regression with `PATH` excluding `paseo`.
  - Requirement: NFR6, p02-t10

- **Section hard-error aggregation loses partial records in the user artifact** (`plugins/consensus/skills/consensus-refine/scripts/consensus-refine.mjs:741`)
  - Issue: If a section fails after one or more successful turns, `runConsensusLoop` has already written `records.json` and `status.json` with the real turn count, but `runSequential` catches the error and replaces that section with `turns: 0`, `rounds: 0`, and `records: []`. The resulting deliberation artifact omits the partial audit trail and reports inaccurate section state, despite the write-through records contract and FR5's requirement that the artifact capture every round's reasoning/verdict.
  - Fix: In the section catch path, load the existing `paths.status` and `paths.records` when present, and use those in the aggregated section result. Preserve the latest revised section output when recoverable; otherwise keep the original section but still render the partial records and error status.
  - Requirement: FR5, p02-t03, p02-t10

## Findings

### Critical

- **Max-rounds resumes record user intervention without continuing deliberation** (`plugins/consensus/skills/consensus-refine/scripts/consensus-loop.mjs:770`)
  - Issue: `runConsensusLoop` calculates `maxTurns` from the absolute `options.maxRounds * options.peers.length`, then starts the peer loop at `peerTurnCount(records)`. On resume, `appendUserIntervention` runs before the loop, but user records do not affect `peerTurnCount`. If the resumed artifact already has `peerTurnCount(records) === maxTurns` from a real `max-rounds` stop, the loop condition at `plugins/consensus/skills/consensus-refine/scripts/consensus-loop.mjs:783` is false, so no peer is invoked after the user direction. A manual reproduction returned `calls: 0`, `status: "max-rounds"`, and records ending in `USER_INTERVENTION`.
  - Fix: Treat resume intervention as a new continuation budget or require/validate an increased budget before accepting `--user-direction`. A pragmatic fix is to set the loop ceiling to `peerTurnCount(initialRecords) + (options.maxRounds * options.peers.length)` when resuming with user direction, or add an explicit `--additional-rounds` continuation budget. Add a regression where the resumed artifact has status `max-rounds`, prior peer turns already equal `maxRounds * peerCount`, and `--user-direction` must invoke the next peer.
  - Requirement: FR6 / p04-t03. The plan requires "continuing from the next agent turn" after a user intervention, and the spec requires max-rounds impasse resume to continue from artifact state.

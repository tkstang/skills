# Discovery: consensus-evaluate

## Initial Request

Implement backlog item **bl-5174 — "Add consensus-evaluate skill (artifact vs rubric)"**.

Add a new `consensus-evaluate` skill to the consensus plugin: two AI peers judge an
artifact against a rubric/spec and deliberate to a unified evaluation, with per-peer
reasoning and any dissent preserved in the deliberation log. It is a **thin wrapper**
over the existing consensus-loop primitive — do not reimplement deliberation logic.

v3 defaults (all overridable by the standard flags):

- cold start: `shared_input`
- iteration: `parallel_revision` (already shipped; verify it is the default here)
- agency: `minimal` (independent judgment is the point — surface every meaningful
  disagreement rather than editorializing)
- mode-specific input: `--rubric <path>`

Each turn, peers produce an **evaluation** (not an artifact edit); "done" is a stable
unified evaluation.

## Solution Space

This request carried two genuine open questions. Both were explored and resolved
through a brainstorming pass before planning.

### Q1 — How does `consensus-evaluate` reuse the shared deliberation engine?

The engine (`consensus-loop.mjs`) currently lives only inside `refine/` and bakes
refine-specific wording into its prompts (`buildParallelTurnPrompt` literally instructs
peers to "revise the section ... put the full resulting section in `proposed_artifact`").
Evaluation needs peers to produce a judgment *about* the artifact, not an edit *of* it.

- **Approach A (Recommended): Shared engine + narrow prompt-profile seam.** Move the
  engine + its schemas to a canonical `shared/consensus-core/`, generate byte-identical
  committed copies into each consumer (`refine/`, `evaluate/`) with a `sync:consensus-core`
  script and a drift-guard test — mirroring the existing `shared/transcript-core/`
  precedent. Add an optional `promptProfile` to `runConsensusLoop`'s `runOptions` so a
  consumer can inject evaluation-framed prompt builders; default to the engine's existing
  builders so refine stays behavior-identical.
- **Approach B: Goal-encoding (stuff the artifact into `--goal`).** Rejected — a genuine
  safety smell, not just aesthetics. The artifact under judgment is untrusted content; the
  engine only wraps the *section* in untrusted-content framing, not the `--goal`. This puts
  untrusted text outside the guardrail, and the prompt would still say "revise the section,"
  inviting peers to edit rather than judge — the exact bug the design question exists to
  prevent.
- **Approach C: Fork the engine into evaluate/.** Rejected — the one thing repo convention
  (the transcript-core drift guard) and the item brief explicitly forbid: "shared … reuse
  it, don't fork it."

### Q2 — What shape should the converged evaluation artifact take?

- **Free-form markdown evaluation with a seeded template (Recommended).** The converged
  markdown document is the "unified findings"; per-peer reasoning + dissent come free from
  the deliberation log (each record already carries `agent`, `reasoning`, `verdict`,
  `critique`, `proposed_artifact`). Reuses `verdict-parallel.schema.json` verbatim —
  `proposed_artifact` simply carries the evaluation doc. No new schema injection, no
  convergence-detection change.
- **Structured JSON evaluation schema.** Rejected for an S-item: requires a second engine
  seam (per-mode schema injection) and more wrapper code, with no acceptance criterion that
  demands it. ~80% of the queryability comes free by having the wrapper seed a lightly
  structured template (per-criterion verdict table + findings + overall verdict/confidence).

### Chosen Direction

**A + free-form markdown evaluation**, with two constraints from the user:

1. Add a **narrow** prompt-profile seam to the engine, keeping refine's current prompts as
   the unchanged defaults (refine must remain byte-for-byte behavior-identical).
2. Per-peer reasoning and dissent remain preserved in the **deliberation log** as the source
   of truth; the final evaluation document summarizes unified findings plus any dissent.

No new verdict schema or JSON contract is added for this item unless implementation/tests
prove it necessary.

**User validated:** Yes — explicit "Lock in A + free-form markdown evaluation."

## Key Decisions

1. **Engine packaging:** Promote the deliberation engine + its `schemas/` to a canonical
   `shared/consensus-core/`, with generated byte-identical committed copies in each consumer
   skill, a `sync:consensus-core` npm script, and a drift-guard test — mirroring
   `shared/transcript-core/`. Schema paths resolve relative to each module's own location
   (`new URL('../schemas/...', import.meta.url)`), so the sync unit is **engine + the schema
   JSONs it references**; every consumer gets its own `schemas/` copy.
2. **Engine seam:** `runConsensusLoop` gains `runOptions.promptProfile = { buildTurnPrompt?,
   buildParallelTurnPrompt?, buildSynthesisPrompt? }`, each optional and resolved as
   `profile.X ?? <existing module builder>`. Builder signatures are unchanged. refine passes
   nothing → behavior-identical.
3. **Evaluation semantics live in the wrapper, not the engine:** the wrapper reads the
   artifact + rubric and builds closures that render them as untrusted-content-framed blocks
   and instruct peers to produce/revise an evaluation. The engine never learns what a rubric
   is.
4. **Output contract:** free-form markdown evaluation seeded from a lightly structured,
   rubric-derived template; reuses `verdict-parallel.schema.json` verbatim. The deliberation
   log is the source of truth for per-peer reasoning and dissent.
5. **Wrapper defaults (all overridable):** `--cold-start shared_input`, `--iteration
   parallel_revision`, `--agency minimal`. `--cold-start independent_draft` is rejected with
   a clear message (consistent with refine, which does not yet support it).
6. **Family status:** ship `consensus-evaluate` as **shipped**, not deferred, in manifests,
   SKILL.md, and READMEs.

## Constraints

- Shipped skill code must be **dependency-free Node stdlib only**, runnable with no install
  step (Paseo is the only external boundary; see DR-002).
- Do not fork shared deliberation logic — canonical source + generated committed copies +
  drift guard, per repo convention.
- Conventional Commits; this is item bl-5174 (e.g. `feat(consensus): add consensus-evaluate
  skill`).
- Must pass `npm test`, `npm run validate`, and `npm run smoke` before claiming done.
- refine must remain behavior-identical (verified by re-running its full suite + the new
  drift guard).
- `oxfmt`/`oxlint` never touch generated copies or agent-instruction files; keep the new
  generated engine copies on the existing exclusion lists alongside the transcript-core
  copies.
